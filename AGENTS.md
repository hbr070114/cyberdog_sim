# Karpathy 编码指南

源自 Andrej Karpathy 关于 LLM 编码缺陷的观察。完整规范参考 [andrej-karpathy-skills](https://github.com/swarmclawai/andrej-karpathy-skills)。

**使用方式：** Codex 会自动读取项目根目录的 AGENTS.md 文件并应用这些规则。

**权衡：** 这些规则偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 编码前思考

**不要假设。不要隐藏困惑。要暴露权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，请问。
- 如果存在多种解释，要展示出来——不要默默选择。
- 如果存在更简单的方法，请说明。在合理时要反驳。
- 如果有什么不清楚的，停下来。说出困惑之处并询问。

## 2. 简洁优先

**最小化代码解决问题。不要投机。**

- 不要添加需求之外的功能。
- 不要为单次使用的代码创建抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要处理不可能发生的情况的错误处理。
- 如果你写了 200 行而本来可以 50 行，请重写。

问自己："一位资深工程师会说这过度复杂吗？"如果是的，简化它。

## 3. 精准修改

**只触碰必须触碰的。只清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有风格，即使你可能会用不同的方式做。
- 如果你注意到无关的死代码，提一下——不要删除它。

当你的更改产生孤立代码时：
- 移除因你的更改而变得未使用的导入/变量/函数。
- 不要删除原本就存在的死代码，除非被要求。

测试方法：每一行更改都应该能追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让它们通过"
- "修复 bug" → "写一个能复现它的测试，然后让它通过"
- "重构 X" → "确保之前和之后测试都通过"

对于多步骤任务，简要说明计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

强有力的成功标准让你可以独立循环。弱标准（"让它工作"）需要不断澄清。
